# SKILLS.md

## 목적

이 문서는 Gongmozip FE 작업에서 반복해서 사용하는 실행 절차를 정리합니다.
패키지 매니저, 브랜치 규칙, 폴더 구조, 검증 명령어 같은 프로젝트 전체 규칙은 `AGENTS.md`를 기준으로 따릅니다.

## Figma 화면 구현

Figma 화면 또는 컴포넌트를 구현할 때 아래 순서로 진행합니다.

1. Figma URL에 특정 노드를 가리키는 `node-id`가 포함되어 있는지 확인합니다.
2. URL에서 `fileKey`와 `nodeId`를 추출해 Figma MCP `get_design_context`로 조회합니다.
3. 선택된 노드가 섹션이면 구현 대상 프레임을 확인한 뒤, 해당 프레임 노드로 다시 `get_design_context`를 호출합니다.
4. Figma MCP가 반환한 React/Tailwind 코드는 참고용으로만 사용합니다.
5. 실제 구현은 현재 프로젝트의 Next.js App Router, TypeScript, Tailwind CSS 구조에 맞춥니다.
6. 새 컴포넌트를 만들기 전에 기존 컴포넌트와 스타일 패턴을 먼저 확인합니다.
7. 반복되는 UI는 `src/components` 또는 관련 기능 폴더 근처에 작은 재사용 컴포넌트로 분리합니다.
8. 백엔드 API가 준비되지 않은 동안에는 mock 데이터를 사용합니다.
9. mock 데이터는 나중에 API 데이터로 쉽게 교체할 수 있도록 화면 컴포넌트와 분리합니다.
10. Figma 모바일 기준 폭인 390px 근처에서 먼저 레이아웃을 확인하고, 이후 데스크톱 폭에서 깨짐이 없는지 확인합니다.
11. 가능하면 작업 완료 전 `pnpm lint`와 `pnpm build`를 실행합니다.

## 채팅 기능 구현

채팅 관련 화면은 아래 순서로 구현합니다.

1. 채팅 목록 화면부터 구현합니다.
2. mock 메시지 데이터를 사용해 기본 채팅방 화면을 구현합니다.
3. 상단 네비게이션, 메시지 리스트, 메시지 아이템, 입력바, 모달, 토스트 같은 공통 UI를 분리합니다.
4. 목록과 기본 채팅방 UI가 안정된 뒤 채팅방 설정과 관리 화면을 구현합니다.
5. 백엔드 계약이 확정되기 전까지 액션 핸들러는 로컬 상태 또는 mock 함수로 처리합니다.
6. mock 데이터 단계에서도 로딩, 빈 상태, 에러 상태가 필요한지 확인합니다.
7. API 계약이 확정되면 mock 데이터를 TanStack Query 기반 서버 상태로 교체합니다.

## API 연동

mock 데이터를 백엔드 API로 교체할 때 아래 순서로 진행합니다.

1. 구현 전에 요청과 응답 계약을 확인합니다.
2. 요청 파라미터와 응답 데이터의 TypeScript 타입을 정의합니다.
3. 서버 상태 관리는 TanStack Query 사용을 우선합니다.
4. Zustand는 서로 떨어진 UI 사이에서 공유해야 하는 클라이언트 상태가 있을 때만 사용합니다.
5. 로딩, 빈 상태, 성공 상태, 에러 상태를 함께 처리합니다.
6. API 유틸은 기존 프로젝트 패턴에 따라 기능 폴더, `src/queries`, `src/lib`, `src/utils` 중 적절한 위치에 둡니다.
7. 새 환경 변수를 추가하면 `.env.example`도 함께 업데이트합니다.
8. 가능하면 작업 완료 전 `pnpm lint`와 `pnpm build`를 실행합니다.

## UI 검증

사용자에게 보이는 UI 작업을 마치기 전에 아래 항목을 확인합니다.

1. 모바일 화면에서 오버플로우나 잘림이 없는지 확인합니다.
2. 데스크톱 폭에서 모바일 중심 레이아웃이 어색하게 늘어나지 않는지 확인합니다.
3. 터치 영역과 버튼 상태가 사용 가능한지 확인합니다.
4. 필요한 경우 로딩, 빈 상태, 에러 상태가 표현되어 있는지 확인합니다.
5. 공통 컴포넌트가 기존 네이밍과 스타일 패턴을 따르는지 확인합니다.
6. `pnpm lint`를 실행합니다.
7. `pnpm build`를 실행합니다.

## PR 준비

PR을 만들거나 준비하기 전에 아래 순서로 확인합니다.

1. 기능 작업이라면 브랜치명이 `feature/기능명` 규칙을 따르는지 확인합니다.
2. 작업 범위와 관련 없는 파일이 수정되지 않았는지 확인합니다.
3. `git status --short`로 변경 파일을 확인합니다.
4. `pnpm lint`를 실행합니다.
5. `pnpm build`를 실행합니다.
6. 변경된 화면, 컴포넌트, mock 데이터, 남은 API 연동 작업을 정리합니다.
