# 프론트엔드 채팅 구현 전에 알아둘 것

채팅 기능은 일반적인 게시글/댓글 API와 다르게, REST API와 WebSocket을 함께 사용합니다. 처음 구현할 때는 "처음 들어갈 때 필요한 데이터는 REST API로 가져오고, 이후 새 메시지는 WebSocket으로 실시간 반영한다"는 구조로 이해하면 쉽습니다.

## 1. REST API와 WebSocket의 역할

### REST API

REST API는 사용자가 채팅방에 처음 들어왔을 때 필요한 정보를 가져오는 데 사용합니다.

- 채팅방 정보 조회
- 팀원 목록 조회
- 이전 메시지 조회
- 권한 없음 처리

예를 들어 채팅방에 입장하면 먼저 아래 API를 호출합니다.

```txt
GET /api/teams/{teamId}/chat-room
GET /api/chat-rooms/{chatRoomId}/messages?size=50
```

### WebSocket

WebSocket은 채팅방에 들어온 이후 새 메시지를 실시간으로 주고받는 데 사용합니다.

- 사용자가 보낸 메시지 전송
- 같은 채팅방에 있는 팀원들의 새 메시지 수신
- 연결 끊김, 재연결 처리

REST API는 "처음 화면을 채우는 용도", WebSocket은 "이후 실시간 업데이트 용도"라고 보면 됩니다.

## 2. STOMP란 무엇인가

WebSocket은 실시간 연결 통로이고, STOMP는 그 통로 위에서 메시지를 주고받는 규칙입니다.

프론트는 보통 아래 흐름을 따릅니다.

1. WebSocket 서버에 연결합니다.
2. 특정 채팅방 경로를 구독합니다.
3. 메시지를 보낼 때 정해진 publish 경로로 전송합니다.
4. 서버가 구독 중인 사람들에게 메시지를 다시 전달합니다.

예시 경로는 아래와 같습니다.

```txt
WebSocket endpoint: /ws/chat
메시지 보내기: /pub/chat/message
메시지 받기: /sub/chat/room/{chatRoomId}
```

## 3. 채팅방 입장 흐름

채팅방 화면은 보통 이 순서로 동작합니다.

1. 사용자가 팀 상세 페이지에서 채팅방으로 이동합니다.
2. 프론트가 채팅방 정보를 조회합니다.
3. 백엔드는 로그인 사용자가 팀원인지 확인합니다.
4. 권한이 있으면 `chatRoomId`, 팀명, 팀원 목록을 내려줍니다.
5. 프론트가 이전 메시지를 조회합니다.
6. 이전 메시지를 화면에 렌더링합니다.
7. WebSocket을 연결합니다.
8. 현재 채팅방을 구독합니다.
9. 사용자가 메시지를 보내면 서버에 전송합니다.
10. 서버에서 전달된 메시지를 화면에 추가합니다.

처음에는 이 흐름을 하나씩 구현하면 됩니다.

## 4. 메시지 데이터에서 중요한 필드

프론트에서 메시지를 예쁘게 보여주려면 최소한 아래 값이 필요합니다.

```ts
type ChatMessage = {
  id: number;
  chatRoomId: number;
  senderId: number | null;
  senderNickname: string | null;
  content: string;
  messageType: 'TALK' | 'SYSTEM' | 'BOT';
  createdAt: string;
};
```

각 필드의 역할은 아래와 같습니다.

- `id`: 메시지 고유값입니다.
- `chatRoomId`: 어느 채팅방 메시지인지 구분합니다.
- `senderId`: 보낸 사람 id입니다. 시스템 메시지는 `null`일 수 있습니다.
- `senderNickname`: 보낸 사람 이름을 표시할 때 사용합니다.
- `content`: 메시지 내용입니다.
- `messageType`: 일반 메시지인지, 시스템 메시지인지 구분합니다.
- `createdAt`: 전송 시간을 표시할 때 사용합니다.

## 5. 메시지 타입별 화면 처리

채팅 메시지는 타입에 따라 다르게 보여줘야 합니다.

### TALK

일반 사용자가 보낸 메시지입니다.

- 내가 보낸 메시지는 오른쪽에 표시합니다.
- 다른 팀원이 보낸 메시지는 왼쪽에 표시합니다.
- 닉네임, 메시지 내용, 시간을 표시합니다.

### SYSTEM

서버가 자동으로 남기는 안내 메시지입니다.

- 가운데 정렬된 안내 문구처럼 표시합니다.
- 말풍선보다는 작은 안내 배지 형태가 자연스럽습니다.
- 예: `팀 매칭이 완료되었습니다.`

### BOT

자동 안내 또는 도우미 메시지입니다.

- MVP에서는 `SYSTEM`과 비슷하게 처리해도 됩니다.
- 나중에 온보딩 안내가 많아지면 별도 디자인으로 분리할 수 있습니다.

## 6. 상태 관리 기준

채팅에서는 서버 상태와 UI 상태를 구분하는 것이 중요합니다.

### TanStack Query로 관리하기 좋은 것

- 채팅방 정보
- 이전 메시지 목록
- 서버에서 받은 메시지 캐시

### Zustand로 관리해도 되는 것

- 현재 입력 중인 메시지
- 채팅 패널 열림 여부
- 임시 UI 상태

서버에서 가져오는 데이터는 TanStack Query를 우선 사용하고, WebSocket으로 받은 새 메시지는 TanStack Query cache에 추가하는 방식이 좋습니다.

## 7. 처음 구현할 때의 추천 순서

처음부터 WebSocket을 붙이면 헷갈릴 수 있으니 아래 순서가 좋습니다.

1. 채팅방 화면 UI를 먼저 만듭니다.
2. 더미 메시지로 말풍선 UI를 확인합니다.
3. 채팅방 정보 조회 API를 연결합니다.
4. 이전 메시지 조회 API를 연결합니다.
5. 메시지 입력창과 전송 버튼 UI를 만듭니다.
6. WebSocket 연결을 붙입니다.
7. 구독으로 새 메시지를 받아 화면에 추가합니다.
8. 메시지 전송을 연결합니다.
9. 연결 실패, 권한 없음, 빈 상태를 정리합니다.

이 순서로 가면 화면, REST API, WebSocket을 한 번에 섞지 않아도 됩니다.

## 8. 꼭 처리해야 하는 상태

채팅은 happy path만 구현하면 실제 사용 중에 금방 깨져 보입니다. MVP라도 아래 상태는 챙기는 것이 좋습니다.

- 채팅방 정보 로딩 중
- 채팅방 접근 권한 없음
- 이전 메시지 로딩 중
- 이전 메시지가 없음
- 메시지 조회 실패
- WebSocket 연결 중
- WebSocket 연결 실패
- WebSocket 재연결 중
- 오프라인 상태
- 메시지 전송 실패

## 9. PWA에서 특히 신경 쓸 부분

PWA는 사용자가 앱처럼 다시 열 수 있기 때문에, 재진입과 네트워크 상태 처리가 중요합니다.

- 앱을 다시 열면 REST API로 채팅방과 이전 메시지를 다시 조회합니다.
- 오프라인이면 메시지 전송 버튼을 비활성화합니다.
- WebSocket은 오프라인에서 유지되지 않으므로 연결 상태를 화면에 반영합니다.
- 푸시 알림은 MVP 이후로 미루는 것이 좋습니다.

## 10. 백엔드와 먼저 맞춰야 하는 것

프론트 구현 전에 백엔드와 아래 항목은 꼭 맞춰야 합니다.

- WebSocket 연결 시 JWT를 어디에 담아 보낼지
- 팀원이 아닌 사용자가 접근하면 어떤 에러 코드를 내려줄지
- 이전 메시지 응답과 실시간 메시지 응답의 필드가 같은지
- 이전 메시지 정렬이 최신순인지 시간순인지
- 메시지 id와 createdAt은 서버에서 생성하는지
- 시스템 메시지의 senderId는 null인지 별도 id인지
- WebSocket 연결 실패와 권한 실패를 프론트가 어떻게 알 수 있는지

## 11. 초반에는 하지 않아도 되는 것

아래 기능은 채팅 MVP 이후로 미뤄도 됩니다.

- 읽음 처리
- 안 읽은 메시지 수
- 이미지 전송
- 파일 전송
- 메시지 삭제
- 메시지 수정
- 답장 기능
- 이모지 반응
- 채팅방 검색
- 푸시 알림

처음 구현에서는 "팀원만 들어올 수 있고, 이전 메시지가 보이고, 실시간으로 텍스트 메시지를 주고받는다"까지만 안정적으로 만드는 것이 목표입니다.
