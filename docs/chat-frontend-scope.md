# 채팅 기능 프론트엔드 MVP 범위

이 문서는 백엔드 채팅 설계가 확정되기 전, 프론트엔드에서 초반에 어디까지 구현할지 정리하기 위한 기준입니다.

## 전제

- 프로젝트는 Next.js App Router 기반 PWA입니다.
- 채팅방은 매칭이 확정된 팀마다 1개씩 생성됩니다.
- MVP에서는 실시간 채팅, 이전 메시지 조회, 팀원 권한 검증 흐름을 우선합니다.
- 읽음 처리, 이미지 전송, 메시지 삭제, 푸시 알림은 후순위로 둡니다.

## 프론트엔드 초반 구현 범위

### 1. 채팅방 진입 흐름

- 팀 상세 페이지 또는 매칭 완료 화면에서 채팅방으로 이동할 수 있는 진입 버튼을 제공합니다.
- `GET /api/teams/{teamId}/chat-room`으로 채팅방 정보를 조회합니다.
- 로그인 사용자가 팀원이 아니거나 접근 권한이 없으면 에러 화면 또는 이전 화면으로 안내합니다.
- 채팅방 제목, 팀명, 팀원 목록을 화면에 표시합니다.

### 2. 이전 메시지 조회

- 채팅방 진입 시 `GET /api/chat-rooms/{chatRoomId}/messages?size=50`으로 최근 메시지를 조회합니다.
- 서버가 내려주는 정렬 기준에 맞춰 메시지를 시간순으로 렌더링합니다.
- 로딩 상태, 빈 메시지 상태, 조회 실패 상태를 제공합니다.
- 새로고침 후 다시 들어와도 이전 메시지가 보이도록 API 조회를 기준으로 화면을 복구합니다.

### 3. 실시간 메시지 송수신

- 채팅방 입장 후 WebSocket STOMP 연결을 맺습니다.
- 서버 구독 경로 `/sub/chat/room/{chatRoomId}`를 구독합니다.
- 사용자가 메시지를 보내면 `/pub/chat/message`로 전송합니다.
- 서버에서 전달된 메시지를 현재 메시지 목록에 추가합니다.
- 연결 중, 연결 실패, 재연결 중 상태를 최소한으로 표시합니다.

### 4. 메시지 입력 UI

- 텍스트 입력창과 전송 버튼을 제공합니다.
- 빈 메시지는 전송하지 않습니다.
- 전송 중에는 중복 전송을 방지합니다.
- 전송 실패 시 사용자에게 실패 상태를 보여주고 재시도 가능 여부를 검토합니다.

### 5. 메시지 표시

- 내 메시지와 다른 팀원의 메시지를 구분해 표시합니다.
- 메시지에는 보낸 사람, 내용, 전송 시간을 표시합니다.
- `SYSTEM`, `BOT`, `TALK` 메시지 타입에 따라 표시 방식을 구분합니다.
- 시스템 메시지는 일반 대화와 다르게 중앙 안내 형태로 표시합니다.

### 6. PWA 고려사항

- 새로고침 또는 앱 재진입 시 채팅방 상태가 API 조회로 복구되어야 합니다.
- WebSocket 연결이 끊겼을 때 재연결 또는 새로고침 안내가 필요합니다.
- 오프라인 상태에서는 메시지 전송을 막고 네트워크 상태를 안내합니다.
- 푸시 알림은 MVP 이후 기능으로 분리하되, 추후 확장을 고려해 알림 관련 로직을 채팅 UI와 강하게 결합하지 않습니다.

## 초반에 만들면 좋은 프론트 구조

- `src/app/.../chat/...`: 채팅방 라우트
- `src/components/chat`: 채팅 UI 컴포넌트
- `src/queries` 또는 기능 폴더 내부: 채팅방 조회, 이전 메시지 조회 query hook
- `src/lib` 또는 `src/utils`: STOMP/WebSocket client 생성 유틸
- 필요 시 `src/stores`: 현재 채팅방 UI 상태처럼 서버 상태가 아닌 값만 관리

서버에서 받은 메시지 목록과 채팅방 정보는 TanStack Query를 우선 사용하고, WebSocket으로 들어오는 실시간 메시지는 query cache에 반영하는 방식이 좋습니다.

## 백엔드와 논의해야 할 항목

### 1. 인증 방식

- WebSocket 연결 시 JWT를 어디에 담아 보낼지 정해야 합니다.
- 예: STOMP connect header, query string, cookie 기반 인증
- 토큰 만료 시 연결이 어떻게 끊기고 프론트가 어떤 응답을 받는지 정해야 합니다.

### 2. 권한 검증 기준

- `GET /api/teams/{teamId}/chat-room`에서 팀원이 아니면 어떤 status code를 내려줄지 정해야 합니다.
- 채팅방 메시지 조회 API에서도 권한 검증을 동일하게 하는지 확인해야 합니다.
- WebSocket 구독과 메시지 전송 시에도 서버가 팀원 여부를 검증하는지 확인해야 합니다.

### 3. 메시지 응답 타입

- REST 이전 메시지 응답과 WebSocket 실시간 메시지 응답의 필드명을 동일하게 맞출지 정해야 합니다.
- 프론트 표시를 위해 최소한 아래 필드가 필요합니다.

```json
{
  "id": 1,
  "chatRoomId": 1,
  "senderId": 3,
  "senderNickname": "채영",
  "content": "안녕하세요!",
  "messageType": "TALK",
  "createdAt": "2026-07-09T18:00:00"
}
```

### 4. 메시지 정렬과 페이지네이션

- 이전 메시지를 최신순으로 받을지, 시간순으로 받을지 정해야 합니다.
- 무한 스크롤이 필요하다면 cursor 방식인지 page 방식인지 정해야 합니다.
- MVP에서는 최근 50개 조회 후 시간순 표시를 추천합니다.

### 5. 메시지 전송 성공 기준

- 클라이언트가 보낸 메시지를 즉시 화면에 임시 표시할지, 서버 응답을 받은 뒤 표시할지 정해야 합니다.
- 메시지 id와 createdAt은 서버 기준 값을 사용할지 정해야 합니다.
- 전송 실패 시 서버 에러 형식을 정해야 합니다.

### 6. 시스템 메시지

- 시스템 메시지를 백엔드가 DB에 저장해서 내려주는지 확인해야 합니다.
- 시스템 메시지의 senderId가 null인지, 별도 bot/system id를 사용하는지 정해야 합니다.
- 메시지 타입 값은 `TALK`, `SYSTEM`, `BOT`처럼 enum으로 고정하는 것이 좋습니다.

### 7. 채팅방 생성 시점

- 팀원 전원 수락 후 `Team` 생성과 동시에 `ChatRoom`이 항상 생성되는지 확인해야 합니다.
- 이미 생성된 팀에 채팅방이 없는 예외 상황이 발생할 수 있는지 확인해야 합니다.
- 채팅방 조회 시 아직 생성 중이면 어떤 응답을 내려줄지 정해야 합니다.

### 8. WebSocket 경로와 이벤트 규칙

- WebSocket endpoint: `/ws/chat`
- publish path: `/pub/chat/message`
- subscribe path: `/sub/chat/room/{chatRoomId}`
- 연결 성공, 연결 실패, 권한 없음, 메시지 전송 실패 이벤트를 어떤 형식으로 처리할지 정해야 합니다.

### 9. 시간과 타임존

- `createdAt`은 ISO string으로 내려주는지 확인해야 합니다.
- 서버 시간대 기준과 프론트 표시 기준을 정해야 합니다.
- 프론트에서는 한국 시간 표시가 필요한지, 상대 시간 표시가 필요한지 확인해야 합니다.

### 10. MVP 이후 기능

- 읽음 처리
- 이미지 또는 파일 전송
- 메시지 삭제
- 채팅방 목록
- 새 메시지 배지
- PWA 푸시 알림
- 오프라인 메시지 임시 저장

위 기능은 MVP 이후로 분리하고, 초반 구조가 확장을 막지 않도록만 고려합니다.
